v0.4 한 페이지 요약 — 이동원 솔로 + 작업별 보고서

v0.2 ROADMAP(4명 분업)이 작업 분담 측면에서는 깔끔했지만, 실제로는 이동원 1인이 전 단계를 끌고 가게 됨. v0.4는 이 현실을 반영하여 모든 단계의 산출물·보고서·버전 관리를 이동원 책임으로 재정렬. 통합 단계(_integrated/)는 추후 별도 1인.

이 ROADMAP의 사용 방식 — 작업 시작 전 해당 단계 카드를 한 번 읽고, 산출물 매트릭스에서 다음 파일명을 확인. 끝나면 위반금지 8항목 self-check + 보고서 마무리.

v0.2 → v0.4 변경의 핵심 4가지

1 · 솔로 워크플로우

4명 분업 → 1인 책임

  • v0.2의 A·B·C·D 역할 매트릭스 deprecate
  • 이동원이 수집·전처리·정제·마이닝 전 단계 작업
  • 모든 작업물은 members/이동원/ 안에
  • 통합은 5/19 이후 별도 1인 (이동원 또는 별도)

2 · 작업별 보고서

교수님 가이드 03 형식
  • 각 노트북당 1개 보고서 (.md)
  • 양식: 목적 / 데이터 / 방법 / 결과 / 한계 / 다음 단계
  • 작성 시점: 셀 단위 메모 (Q4-a) + 마무리 정리 (Q4-c)
  • 파일명: {NN}_{단계}_phase{N}_v{M}_보고서.md

3 · 노트북 버전 명시

v→v+1 모두 보존
  • 명명: {NN}_{단계}_phase{N}_v{M}.ipynb
  • 예: 01_collection_phase1_v1.ipynb → v2 → v3
  • 사소한 수정(주석·변수명)도 새 파일 (rule 1: 버전 보존)
  • 구버전 삭제 ❌ — 결과 안 좋으면 rollback

4 · 웹크롤링·벤치마킹 지속

작업마다 1건 이상
  • 각 작업마다 외부 코드 1건 + 학술 논문 1건 이상 확인
  • 벤치마킹 결과는 보고서 "방법" 섹션에 인용
  • v0.2 COLLECTION 4-track 벤치마킹 그대로 유지 (강의·공개코드·실무·논문)
  • 웹크롤링은 가이드 매트릭스 안에서만 (DART API 주력)

작업별 보고서 양식 — 교수님 가이드 03 형식

각 노트북 1개당 보고서 1개. 양식은 docs/professor_md/03_minimal_analysis_example.md를 참고. 작성 시점은 셀 작업 중 메모(Q4-a) + 노트북 끝난 후 정리(Q4-c) 병행.

표준 양식 (.md 6 섹션)

# {NN}_{단계}_phase{N}_v{M} — {짧은 제목}

> **작성자**: 이동원
> **작성일**: YYYY-MM-DD
> **노트북**: `members/이동원/notebooks/{NN}_{단계}_phase{N}_v{M}.ipynb`
> **노트북 버전**: v{M}
> **이전 버전**: archive 또는 v{M-1}

## 1. 목적 (Why)

이 작업이 왜 필요한가? 분석 가설과 어떻게 연결되는가? 가이드 어느 단계 / 어느 문장에 해당하는가?
- 가이드 인용: `docs/professor_md/02_dart_data_collection.md` §N "..."
- 분석 가설 연결: ESG 언어 → KCGS 등급 (전처리 quality가 천장)

## 2. 데이터

- 입력: {경로 + 형식 + 행수 + 컬럼}
- 출력: {경로 + 형식 + 행수 + 컬럼}
- gitignore 정책: raw/interim/processed는 gitignore. 코드/메타만 git에.

## 3. 방법 (What & How)

단계별로 무엇을 했는지 + 왜 그 방법을 선택했는지 + 벤치마킹 출처.

### 3.1 단계 A
- What: ...
- Why: ...
- 근거: 교수님 가이드 §N / 공개 코드 dart-fss / 학술 Li 2021

## 4. 결과
- 정량: 행수, 시간, 성공률, 시각화 1~2개 첨부
- 정성: 예상과 달랐던 점, 발견한 패턴

## 5. 한계 · 위험
- 알려진 한계 (예: 특정 firm-year의 보고서 양식이 달라 정규식 일부 누락)
- 다음 버전에서 개선할 점
- 위반금지 8항목 self-check 결과

## 6. 다음 단계
- 다음 노트북: `notebooks/{NN+1}_*_v1.ipynb`
- 이 작업의 산출물을 어떻게 받아가는가
- 검증 필요 항목

작성 시점 — Q4-a + c 병행

Q4-a · 셀 작업 중 메모

  • 각 셀 작업 끝날 때마다 보고서에 1~2줄 추가
  • 특히 "왜 이 방법을 선택했는가" 메모 (잊기 전에)
  • 실패한 시도도 메모 → "5. 한계"에서 발전
  • 벤치마킹 출처 (확인한 외부 코드/논문) 즉시 기록

Q4-c · 노트북 끝난 후 정리

  • 전체 흐름·결과를 6 섹션 양식으로 재구성
  • 시각화 추가 (1~2개)
  • 위반금지 8항목 self-check 표 추가
  • 다음 단계 명시 (다음 노트북 이름 + 의존성)
보고서 없는 노트북 commit 금지 — 노트북·보고서는 1:1 매칭. PR description에 보고서 .md 경로도 함께 명시.

노트북 버전 관리 — v→v+1 모두 보존

사소한 수정도 새 파일. 결과 안 좋으면 rollback. v0.3 archive 컨벤션을 더 강화.

명명 규칙

{NN}_{단계}_phase{N}_v{M}.ipynb

  • NN — 단계 번호 (01 수집 / 02 전처리 / 03 마이닝 / 04 통합)
  • 단계 영문명 — collection / preprocessing / mining / integration
  • phaseN — 단계 내 sub-phase
  • vM — 버전 번호 (v1 → v2 → ...)

v→v+1 전환 기준

✅ 새 v 만들어야 할 때

  • 가이드의 새 단계 추가
  • 알고리즘·라이브러리 변경 (Okt → Mecab)
  • 파라미터 sweep 결과 보존 (vocab 5k → 10k)
  • 실패한 시도라도 학습 가치 있음

⚠️ 모호한 케이스

  • 오타·변수명 수정만 → git history로 충분 (선택)
  • 마크다운 보강만 → git history로 충분 (선택)
  • 출력만 다시 실행 → 새 v로 (다른 결과 보존)
