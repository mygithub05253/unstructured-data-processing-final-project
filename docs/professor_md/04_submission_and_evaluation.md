# ESG DART 발표와 제출 가이드

분석이 끝났다면, 이제 결과를 정리하고 발표할 차례입니다. `02_dart_data_collection.md`에서 수집한 데이터와 `03_minimal_analysis_example.md`에서 실행한 분석을 바탕으로, 이 문서는 무엇을 제출하고 어떻게 평가받는지 안내합니다.

## 1. 제출물

| 제출물 | 형식 | 내용 |
|---|---|---|
| 분석 노트북 | `.ipynb` | 데이터 수집, 전처리, feature 생성, 회귀분석 코드와 출력 |
| 보고서 | `.md` | 데이터·방법·결과·한계 순으로 서술 |

노트북은 코드와 표가 실행 가능한 상태여야 합니다. 보고서는 데이터·방법·결과·한계 순으로 쓰되, 각 방법을 왜 선택했는지가 드러나야 합니다.
- **형식**: 마크다운(`.md`)으로 작성합니다. Marp 등으로 발표 자료로 변환할 수 있어 보고서와 발표 준비를 동시에 진행할 수 있습니다.
- **서술 방식**: 3문장 이상의 내용은 첫 문장을 핵심 주장 산문으로, 이후 내용은 bullet으로 정리합니다.

## 2. 보고서에 드러나야 할 판단

아래 흐름은 산출물 목록이 아니라 판단 목록입니다. 표와 코드를 제출하는 것에서 끝나지 않고, 팀이 무엇을 선택했고 왜 그렇게 선택했는지를 설명합니다.

- [ ] 어떤 381개 기업-연도를 사용했는가 (표본의 구성과 한계, timing: `esg_year = fiscal_year + 1`)
- [ ] 보고서 못 찾은 행, 텍스트 추출 실패 행, 결측 feature를 어떻게 처리했는가
- [ ] 사업보고서의 어느 섹션(II, IV, VI)에서 문단을 뽑았고, 왜 그 섹션을 선택했는가
- [ ] 어떤 feature(seed TF-IDF, expanded dictionary, fastText 후보/feature, 기준 문장 cosine similarity, 보고서 위치별 점수)를 만들었는가
- [ ] 각 feature가 무엇을 측정한다고 보는지 쉬운 말과 예시로 설명했는가
- [ ] expanded dictionary를 seed 10개씩에서 출발해 embedding으로 후보를 넓히고, 팀 기준으로 걸러낸 사전으로 설명했는가
- [ ] theta, 후보 단어 직접 검토·걸러내기, 불용어·회사명 제거 기준을 어떻게 정했는가
- [ ] 상위/하위 문장 예시를 직접 읽고 feature 해석이 맞는지 확인했는가
- [ ] 회귀분석 전에 feature와 ESG 등급의 Spearman 순위상관, t-test 또는 Mann-Whitney 검증을 보고하고 그 결과로 feature를 판단했는가
- [ ] OLS, ordered logit, binary logit 중 무엇을 사용했고 왜 그 모델을 선택했는가
- [ ] 결과(계수 방향과 크기)를 어떻게 해석했는가
- [ ] cheap-talk 가능성을 검토했는가 — 단어 빈도가 강한 ρ를 보였다면, 그것이 진짜 ESG 성과 측정인지 단순 disclosure verbosity(공시 장황함)인지 구분해 논의했는가
- [ ] 심화 분석(알파 분석 — 기본 과제 외 팀이 자체 설계한 추가 검증)을 무엇으로 정했고, 그 선택이 무엇을 더 확인하게 해주었는가
- [ ] 왜 인과관계가 아니라 연관성으로 조심스럽게 읽어야 하는가

## 2-1. 제출 전 확인 체크리스트

제출물은 **분석 노트북(.ipynb)과 보고서(.md)** 입니다. 제출 전 아래 항목을 팀이 함께 확인합니다.

### 데이터 lineage
- [ ] 최대 381행 기준으로 수집을 시도했다
- [ ] 각 행에 `corp_code`, `rcept_no`, `fiscal_year`가 올바르게 연결되어 있다
- [ ] 수집 불가 행이 있으면 최종 보고서에 이유를 적었다
- [ ] 분석 corpus와 회귀 패널은 수집·추출 성공 행으로 만들었고, 실패 행은 가짜 0으로 채우지 않았다

### 분석 재현성
- [ ] 노트북을 위에서 아래로 순서대로 실행하면 결과가 재현된다
- [ ] API key가 코드에 직접 포함되어 있지 않다 (환경변수 `.env`만 사용)

### 보고서 판단 기록
- [ ] 사용한 섹션(II/IV/VI)과 문단 추출 기준을 설명했다
- [ ] feature 선택(seed/expanded/fastText) 이유와 한계를 적었다
- [ ] 회귀모형 선택(OLS/ordered logit/binary logit) 이유를 적었다
- [ ] 결과를 인과관계가 아닌 연관성으로 조심스럽게 해석했다

## 3. 평가에서 보는 것

| 영역 | 좋은 결과물의 모습 | 흔한 문제 |
|---|---|---|
| 데이터 이해 | 381개 기업-연도 표본의 의미, 누락 행 처리, timing 기준을 설명한다. | 일부 기업만 골라 전체 결과처럼 말하거나 결측 처리 기준을 숨긴다. |
| DART lineage | `stock_code`, `corp_code`, `rcept_no`, `fiscal_year`를 구분하고 추출 로그를 남긴다. | 회사명만으로 merge하거나 접수번호가 다른 보고서를 섞는다. |
| 텍스트 처리 | II/IV/VI 섹션 발췌, 불용어 제거, seed/expanded/fastText feature의 차이와 선택 기준을 설명한다. | 보고서 전체를 그대로 넣거나, fastText 후보를 검토 없이 ESG 사전으로 확정한다. |
| 측정 validity | 각 feature가 KCGS 등급과 통계적 연관(Spearman 순위상관, 등급 그룹 평균 차이 t-test 또는 Mann-Whitney)을 보고하고 그 결과로 feature를 정당화한다. | feature 분포만 보고 grade와의 관계 검증 없이 회귀로 직행한다. |
| 모형 선택 | OLS, ordered logit, binary logit 중 팀 질문에 맞는 모형을 고르고 버린 대안의 한계를 설명한다. | 모형 선택 근거 없이 OLS 결과만으로 등급 변화·인과 단정 |
| 해석과 한계 | 계수 방향, 효과 크기, cheap-talk 가능성, 업종·표본 한계를 함께 읽는다. | 유의성만 보고 실제 ESG 성과나 인과효과로 단정한다. |
| 알파 | 팀이 스스로 찾은 추가 분석 또는 검증의 이유, 결과, 한계를 설명한다. | 어려운 기법 이름만 붙이고 왜 필요한지 설명하지 않는다. |

Spearman 순위상관은 "feature 값이 큰 기업-연도일수록 등급 순위도 높은가"를 보는 검증입니다. ESG 등급은 서열변수이므로, 보고서에는 아래 형식의 표처럼 feature별 판단을 남깁니다.

| feature | grade | Spearman ρ | p-value | 판단 |
|---|---|---:|---:|---|
| expanded_tfidf_score | ESG grade | 0.15 | 0.004 | 양의 관계가 있으나 크기는 약함 |
| seed_tfidf_score | ESG grade | 0.03 | 0.55 | 뛚렬한 관계 없음 |

좋은 해석은 "ρ가 양수이고 p-value가 작으므로 등급과 양의 연관이 있다"에서 멈추지 않습니다. ρ가 0.10-0.20 수준이면 통계적으로 유의해도 효과는 약하므로, 문장 예시와 cheap-talk 가능성을 함께 확인해야 합니다.

재현성 관련: 노트북 실행 순서, 출력 표, API key 제거 상태가 확인되어야 합니다.

## 3-1. 알파 영역

알파는 팀이 스스로 찾아 추가한 분석 또는 검증입니다. 기법이 어려울수록 높은 점수를 받는 영역이 아닙니다. 좋은 알파는 "왜 이것이 필요한가", "무엇을 더 확인했는가", "어떤 한계가 남았는가"가 분명합니다.

예를 들어 업종별로 결과를 나누어 보거나, 보고서 위치별 점수(section-specific feature)를 비교하거나, cheap-talk 통제를 다르게 두는 방식이 가능합니다. 이 예시는 출발점일 뿐이며, 팀의 질문에 맞는 다른 검증을 선택해도 됩니다.

## 4. 해석할 때 조심할 점

1. 381행은 수업용 pilot panel입니다. KCGS 전체 기업의 대표표본이 아닙니다.

   모든 팀은 최대 381행을 기준으로 수집을 시도합니다. 분석 corpus와 회귀 패널은 그중 수집·추출에 성공한 행으로 만들며, 실패 행은 최종 보고서에 이유를 적고 텍스트 점수 0으로 강제하지 않습니다.

2. 사업보고서 표현은 실제 ESG 성과와 같지 않습니다. 어떤 기업은 실제 성과가 좋아서 많이 쓸 수 있고, 어떤 기업은 설명이 필요한 이슈가 있어서 많이 쓸 수도 있습니다.

3. 기준 문장 cosine similarity는 기준 문장과 공시 문장의 텍스트상 가까움을 보는 보조 측정값입니다. 실제 ESG 성과나 등급을 자동 판정하는 심판이 아닙니다.

4. ESG 등급은 D~A+의 서열변수입니다. 팀의 질문에 따라 OLS, ordered logit, A 이상 여부 binary logit 중 하나를 선택해 사용합니다. OLS는 직관적인 baseline이지만 등급 간 간격이 같다는 가정을 둡니다. ordered logit은 등급 순서를 보존하지만 계수 해석이 어렵고, binary logit은 A 이상 여부를 쉽게 해석하는 대신 등급 정보를 줄입니다. 계수 크기를 등급 점수 차이로 단순 환산하지 마세요.

5. 이 과제의 타이밍 기준은 **`esg_year = fiscal_year + 1`**입니다. 즉 KCGS 평가연도 t의 등급과 직전 회계연도 t-1 사업보고서 언어가 어떻게 정렬되어 있는지 서술적으로 확인합니다. 이는 등급 예측, 공시의 인과효과, 엄밀한 시점 식별이 아닙니다. KCGS 발표일과 사업보고서 제출일의 세부 timing은 이 과제 범위에서 단정하지 않습니다.

## 5. API key 제거 확인

제출 전에 노트북과 보고서에서 API key 값이 보이지 않는지 확인합니다. 환경변수(`.env` 파일)에만 저장하고, 코드에 직접 쓰지 않습니다.
