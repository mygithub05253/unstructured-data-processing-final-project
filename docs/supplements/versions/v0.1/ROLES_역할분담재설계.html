역할 분담 재설계 — 4명 전원 참여 모델

README의 A·B·C·D 분담을 "한 사람이 한 분야"에서 "모두가 전 단계, 단계별 lead 회전"으로 재설계. 개인 폴더 병렬 작업 + AI 병합 방식의 실무적 평가도 함께 정리했습니다.

팀 인원 4명 모델 3-Phase (탐색→생산→통합) 제출물 .ipynb + .md 버전 v0.1

한 페이지 요약

"4명이 각자 폴더에서 전체 파이프라인을 시도하고 AI로 비교·병합한다"는 사용자 제안을 검토한 결과, 탐색 기간에는 매우 좋고, 생산 기간에는 일부 수정이 필요합니다.

결론 한 줄 — Phase 1(탐색): 4명이 각자 폴더에서 1-2 firm-years 파일럿. Phase 2(생산): 단계별 lead/co-lead로 효율화. Phase 3(통합): 1명이 master 정리.

핵심 원칙 4가지

🎯 원칙 #1

모두가 전 단계를 경험한다

한 명이 수집만, 다른 사람이 회귀만 하면 학습 효과가 비대칭. 모든 팀원이 수집·전처리·feature·회귀를 최소 1번씩 손으로 만져봅니다.

🎯 원칙 #2

단계별 lead 회전

전원 참여하되 각 단계에 "이번 단계 책임자(lead) 1명 + co-lead 1명"을 둬서 결정 정체를 막습니다. 4단계마다 lead가 바뀌어 4명이 한 번씩 책임을 짊어집니다.

🎯 원칙 #3

개인 작업은 탐색 단계에서만 병렬

파일럿(1-2 firm-years)은 각자 폴더에서 4명이 독립 시도. 본격 생산(381 행)으로 가면 코드를 하나로 합치고 PR 모델로 전환.

🎯 원칙 #4

GitHub master 1명 병행

PR 리뷰·머지·브랜치 정리·릴리스 노트는 한 사람이 병행. 다른 책임은 똑같이 부담하되 "조율 책임"만 추가.

3-Phase 모델 미리보기

Phase 1 (1-2주차) · 탐색

4명이 각자 폴더에서 1-2 firm-years 파일럿

개인 폴더 병렬 작업 + AI 비교 → 가장 깔끔한 접근법 발견. 사용자 제안한 모델이 가장 잘 작동하는 구간.

Phase 2 (3-5주차) · 생산

단계별 lead/co-lead로 381행 본격 수집·분석

공용 src/, notebooks/에서 작업. 병렬 작업 → PR 모델로 전환. 알파 분석만 개인 자유 영역.

Phase 3 (6주차+발표 직전) · 통합

최종 노트북·보고서를 한 명이 master로 정리

개별 결과를 모아 최종 .ipynb + .md 1세트로. 나머지 3명은 동시에 발표 자료·QnA 준비.

README 기존 분담 vs 재설계 모델

기존은 한 명이 한 분야를 책임지는 "Silo 모델". 재설계는 모두가 전 단계를 거치되 단계별로 lead를 회전시키는 "Round-robin 모델".

기존 README 분담

이름주 책임접근법
A코디네이션 · GitHub master전반 조율, PR 관리
B데이터 수집OpenDART API, corp_code 채우기
C전처리 · Feature토큰화, TF-IDF, fastText
D회귀 · 알파 분석OLS/ordered logit, 알파
기존 모델의 위험
  • 학습 효과 비대칭 — B는 회귀를 모르고, D는 수집을 모름. 발표 때 다른 영역 질문에 답을 못 함.
  • 단일 장애점 — B가 막히면 수집이 멈추고, C가 막히면 feature가 멈춤. 다른 사람이 도울 손이 없음.
  • 비정형 데이터·마이닝 강조가 교수님 평가 핵심인데, 한 명만 텍스트 마이닝을 깊게 봄.

재설계 모델 — Round-robin lead

단계LeadCo-leadReviewer

2. 원문·섹션 추출BAC, D

회전 규칙 — 4명이 각각 lead 2회 + co-lead 2회씩 가져갑니다. 모든 단계에 4명 다 참여하되, lead는 의사결정 책임, co-lead는 검증, reviewer는 PR 확인. 회의는 단계 시작·종료 시 짧게(15분).

  • Co-lead: 원문 추출, 검증

B

  • Lead: 원문 추출, 알파
  • Co-lead: 수집, fastText
  • collection_log 관리

C

텍스트 처리 강함
  • Lead: 전처리, fastText
  • Co-lead: TF-IDF, 알파

D

통계·해석 강함
  • Lead: TF-IDF, 검증
  • Co-lead: 전처리, 회귀
  • 해석 가드레일 관리

사용자 제안 검토 — "개인 폴더 병렬 + AI 병합"

제안 내용: "사람마다 폴더를 생성해서 각각 작업하면서 AI한테 비교하면서 병합한다." 실무적 관점에서 평가합니다.

이 모델을 학원에 비유하면, 4명이 같은 문제를 각자 풀고 → 강사가 답안을 비교해서 가장 좋은 풀이를 채택하는 방식입니다. 학습 효과는 매우 크지만, 만약 매주 4명 모두가 모든 문제를 풀면 시간이 4배 드는 문제가 있습니다. "어떤 단계에서 이걸 할지"가 핵심.

✅ 장점 (이 모델이 빛나는 순간)

장점 #1

학습 효과 극대화

4명이 각자 전 단계를 끝까지 만져봄. 발표 때 어떤 질문이 와도 답할 수 있는 깊이.

장점 #2

"가장 잘된 부분 병합"으로 최종 품질↑

사람마다 강점이 다름. A의 수집 코드 + C의 전처리 + D의 회귀 해석을 골라 모으면 평균을 넘는 결과.

장점 #3

다양한 접근 → 알파 분석 풍부

4명이 다른 θ, 다른 섹션 조합, 다른 모형을 시도하면 자연스레 알파 분석 후보가 4개 이상 쌓임.

장점 #4

Git conflict 위험 ↓ (탐색 단계)

개인 폴더면 충돌 자체가 일어날 일이 없음. 머지를 한 번에 의도적으로 함.

⚠️ 위험 (이 모델이 무너지는 순간)

위험 #1

노동량 4배

파일럿 1-2개는 괜찮지만, 381행 수집을 4명이 각자 하면 OpenDART rate limit + 시간 측면에서 비효율. 특히 수집 단계는 한 번만 깔끔하게 끝내는 게 맞음.

위험 #2

병합 시점에 root cause 추적 어려움

A의 수집 + C의 전처리 + D의 회귀를 합쳤는데 결과가 이상하면 어느 단계의 어떤 선택 때문인지 추적이 힘듦. 단일 master가 더 디버깅 쉬움.

위험 #3

4명의 raw 데이터가 같은지 검증 필요

각자 corp_code·rcept_no를 채웠는데 만약 다르면? 분석 corpus가 다르면 비교가 불가능. 공용 master CSV로 통일해야 함.

위험 #4

AI 병합의 환각 위험

AI가 4명 코드를 비교해주는 건 좋지만, 미묘한 버그·통계 가정 차이를 놓칠 수 있음. 사람이 최종 결정해야 함.

💡 실무 표준 모델과 비교

모델실무에서 쓰는 상황이번 프로젝트 적합도
개인 fork → 비교 머지 (사용자 제안)오픈소스 초기 prototype, 헤커톤 첫날🟢 파일럿 단계 매우 적합 / 🔴 생산 단계 비효율
Feature branch + PR (GitHub flow)대부분의 회사·팀 프로젝트🟢 생산 단계 표준. 본 레포가 이미 이 구조로 셋업됨
Trunk-basedCI/CD 성숙도 높은 팀🔴 이번 프로젝트엔 과함
1명 master + 3명 검토긴급 hotfix, 단일 작성자 문서🟡 통합 단계에만 적합
결론 — 사용자 제안은 "틀린 게 아니라 시점이 중요한 모델"입니다. Phase 1(탐색)에서는 적극 활용, Phase 2(생산)부터는 PR 모델로 갈아탑니다. 다음 탭의 3-Phase 모델이 권장안입니다.

3-Phase 권장 모델 — 탐색·생산·통합

사용자의 "개인 폴더 + AI 병합" 아이디어를 살리되, 단계별로 협업 방식을 다르게 가져갑니다.

Phase 1 · 탐색 (M0~M1 · 1-2주)

4명이 각자 폴더에서 파일럿 (1-2 firm-years)

사용자 제안 모델이 가장 잘 작동하는 구간. 각자 자신의 방식으로 end-to-end를 끝까지 통과시켜 봅니다.

구체 활동
  • 각자 notebooks/individual/{이니셜}/ 폴더에서 작업
  • 같은 firm-year(예: 005930-2024) 1-2개로 수집→전처리→TF-IDF→회귀까지 끝까지
  • 매주 금요일 30분 회고: 4명의 접근법 비교, AI(Claude/ChatGPT)에 코드 diff 보여주고 차이점 정리
  • 산출물: 4명 각자의 파일럿 노트북 4개 + Phase 1 회고 보고서 1개
이 단계의 목표 — 결과보다 학습. 각자가 어디서 막혔고, 어떤 선택을 했는지를 비교해서 Phase 2의 표준 결정.
Phase 2 · 생산 (M1~M3 · 3-5주)

단계별 lead/co-lead로 381행 본격 작업

개인 폴더 시대를 끝내고 공용 src/, notebooks/로 통합. 단계별 lead가 PR을 주도합니다.

구체 활동
  • 각 단계 lead가 feature/{단계}-{이슈} 브랜치에서 작업
  • co-lead는 매일 코드 리뷰 (PR 시 별도 의견)
  • 나머지 2명은 reviewer (위반금지 체크리스트 + 결과 합리성 확인)
  • 알파 분석은 개인 자유 영역 — experiment/{이니셜}-{주제} 브랜치에서 각자 시도
  • 산출물: 통합 notebooks/01_collection.ipynb, 02_preprocessing.ipynb, 03_features.ipynb, 04_regression.ipynb + 4명의 알파 노트북
Phase 2에서 사라지는 것 — 개인 폴더(notebooks/individual/)는 archive로 이동. 모든 새 코드는 공용 브랜치에.
Phase 3 · 통합 (M3~M4 · 1주)

최종 .ipynb + .md를 한 명이 master로 정리

통합 노트북 4개를 하나의 제출용 .ipynb로 합치고, 알파 분석 1개를 선별해 보고서에 추가합니다.

알파 분석은 개인 자유 영역

A의 알파 후보

  • 섹션별(II/IV/VI) 점수 분리 회귀
  • 업종별(KRX 분류) 결과 비교

B의 알파 후보

  • 정정공시 vs 원공시 표현 차이
  • 접수일 ↔ 등급 발표일 timing 분석

C의 알파 후보

  • dense embedding(SBERT) cosine
  • 토픽모델링(LDA) 보조

D의 알파 후보

  • cheap-talk 변수 통제 강화
  • E·S·G 별도 회귀로 차원 분해

폴더 구조 — Phase 1과 Phase 2의 차이

사용자 제안한 "사람마다 폴더 생성" 모델은 Phase 1에서만 적용. Phase 2부터는 공용 폴더로 합칩니다.

Phase 1 (탐색) — 개인 폴더 병렬

unstructured-data-processing-final-project/
├── notebooks/
│   └── individual/                  ← Phase 1 전용 (개인 자유 영역)
│       ├── A/
│       │   ├── 01-A-pilot-005930-2024.ipynb
│       │   └── notes-A.md
│       ├── B/

│       ├── C/

│           └── 01-D-pilot-005930-2024.ipynb

충돌 없음 — 각자 독립 폴더라 git 충돌이 거의 안 일어남. 매주 회고 회의에서 4개 노트북을 한꺼번에 비교.

CONTRIBUTING.md에 정의된 흐름을 3-Phase 모델에 맞게 풀어 씁니다.

Phase 2 — feature 브랜치 + PR

main
 ├── feature/preprocessing-okt-tokenize    # Lead: C, Co: D

PR 흐름:
 1. lead가 branch 따서 작업
 2. co-lead가 매일 review
 3. PR 시 reviewer 2명도 위반금지 체크리스트 확인
 4. 머지 (squash merge 권장) → 브랜치 삭제

PR 시 위반금지 체크리스트 (CONTRIBUTING.md 발췌)

  • [ ] 실패 행을 가짜 0으로 채우지 않았다

  • [ ] esg_year = fiscal_year + 1 고정

Quick Start — 오늘 당장 할 일

팀 4명이 이 HTML을 같이 보면서 합의해야 할 항목들.

1. 합의 사항 (회의 30분)

  • [ ] 3-Phase 모델 채택 여부 (또는 다른 모델)
  • [ ] 4명의 lead 회전표 확정 (8단계 × 4명)
  • [ ] GitHub master 1명 확정 (PR 머지 권한)
  • [ ] 마일스톤 M1~M4 날짜 (README의 빈 칸 채우기)
  • [ ] Phase 1 파일럿 firm-year 선택 (예: 005930-2024 1개 공통 + 각자 1개 추가)
  • [ ] 매주 회의 시간·요일 (Phase 1 회고 ~30분 + Phase 2 짧은 standup)

2. 환경 셋업 (각자)

git clone https://github.com/mygithub05253/unstructured-data-processing-final-project.git
cd unstructured-data-processing-final-project

python -m venv .venv
.venv\Scripts\activate                    # Windows
# source .venv/bin/activate                # Mac/Linux

pip install -r requirements.txt
pip install nbstripout
nbstripout --install

cp .env.example .env                        # .env에 OPENDART_API_KEY 입력

3. Phase 1 시작 (각자)

# 각자 자기 폴더 만들기
mkdir -p notebooks/individual/A           # 자기 이니셜로

# experiment 브랜치 따기
git checkout -b experiment/A-pilot-005930-2024

# 노트북 생성
touch notebooks/individual/A/01-A-pilot-005930-2024.ipynb

# 작업 → 커밋 → push
git add notebooks/individual/A/
git commit -m "experiment: A pilot 005930-2024 수집 시도"
git push -u origin experiment/A-pilot-005930-2024
