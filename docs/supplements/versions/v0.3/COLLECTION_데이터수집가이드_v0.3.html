새 버전을 만들 때 기존 노트북을 덮어쓰기 X. 결과가 나빠질 때 복구 가능해야 함.

2 출력 보존

GitHub commit 시 모든 cell 출력 유지. nbstripout 사용 ❌.

  • 주의: API key 등 비밀값은 코드 레벨에서 마스킹 필수 (다음 규칙)

출력 보존 정책 — Why & How

규칙 2의 근거와 운영 방법.

왜 출력을 보존하는가

재실행 비용 회피

  • Phase 2 수집 (~380 firm-years): 30~45분
  • 전처리 (KoNLPy 형태소): 30분~2시간
  • 임베딩 학습 (fastText 등): 수십 분
  • 출력 유지 시 PR 리뷰·발표 자료 작성 시 재실행 0초

결정 추적성

  • "이 결정의 근거 출력"이 같은 노트북에 있음
  • "왜 vocab=10k가 아니라 5k?" → 같은 노트북에 두 alt 결과 모두
  • 발표 시 reviewer 질문에 즉답 가능

비교 가능성

  • v1 / v2 / archive 노트북을 나란히 펴서 결과 차이 즉시 확인
  • git diff로 코드만 보던 시절보다 훨씬 명확

보고서·회의 효율

  • 스크린샷 캡처 위치가 GitHub에 그대로
  • 슬랙/회의에 GitHub 노트북 링크 공유만으로 결과 전달

어떻게 출력을 유지하나

반드시 — 출력 보존 절차
  1. Jupyter에서 노트북 실행 → 출력 생성
  2. 저장 (Ctrl+S 또는 메뉴) → .ipynb 파일에 출력 직렬화됨
  3. nbstripout 실행 ❌ — 자동 hook도 ❌
  4. git add notebooks/01_collection.ipynb
  5. git commit -m "feat: collection 완료 — 50건 sample (성공률 96%)"
  6. git push → PR → squash merge

주의 — 출력 보존의 부작용

주의 — 다음 정책으로 대응
  • API key 노출 — 출력에 key가 들어가지 않게 코드에서 처리 → 다음 탭 "API key 안전 (강화)" 참조
  • 파일 크기 증가 — wordcloud·시각화 PNG가 base64로 들어감. 100MB 넘으면 git LFS 고려 (현재는 미사용, raw 데이터는 gitignore이므로 노트북 자체는 보통 5~10MB)
  • 병합 충돌 — 동시 편집 시 출력 셀에서 conflict 다발. 대응: members/{이름}/ 개인 폴더로 격리 + 통합은 _integrated/에서 수동 선택
  • 고객·실명 등 민감 정보 — 본 프로젝트는 공시 데이터만 다루므로 해당 없음. 단, KCGS 등급 원본 출력 시 회사명/등급 그대로 (공시이므로 OK)

API key 안전 (강화) — 출력 보존 환경 대응

출력이 git에 보존되므로 v0.2의 nbstripout 안전망이 사라짐. 코드 레벨에서 마스킹이 유일한 보호선.

반드시 지킬 4 가지

🔴 반드시 — API key 노출 0건 보장
  1. .env에 key 저장, .gitignore에 .env 등재 (이미 적용됨)
  2. print(response.url) 금지 — URL params에 crtfc_key가 들어감
  3. print(response.text) 시 mask_key() 통과 의무
  4. requests 호출 시 params=로 전달 (URL 생성 자동, 직접 문자열 조립 ❌)

표준 패턴 — 복사해서 사용

1) key 로딩

from dotenv import load_dotenv
import os

load_dotenv()
API_KEY = os.getenv("OPENDART_API_KEY")
assert API_KEY, ".env에 OPENDART_API_KEY 설정 필요"

2) mask_key 헬퍼

def mask_key(text: str) -> str:
    """문자열에서 API key를 <MASKED>로 치환. 출력 전 반드시 통과."""
    if not API_KEY or not text:
        return text
    return text.replace(API_KEY, "<MASKED>")

3) requests 호출 (URL 노출 X)

response = requests.get(
    "https://opendart.fss.or.kr/api/list.json",
    params={
        "crtfc_key": API_KEY,
        "corp_code": corp_code,
        "bgn_de": "20240101",
    },
)
# response.url 출력 금지!
# 디버깅 필요 시:
print("Status:", response.status_code)
print("URL 마스킹:", mask_key(response.url))   # ✅ 안전
print("본문:", mask_key(response.text[:500]))

4) 에러 메시지도 마스킹

try:
    response.raise_for_status()
except requests.HTTPError as e:
    # 에러 메시지에 URL이 들어갈 수 있음
    print(mask_key(str(e)))
    raise

점검 체크리스트 — 커밋 직전 확인

  • ☐ git diff notebooks/01_collection.ipynb | grep -i crtfc_key 결과 무 (key 패턴 비노출)
  • ☐ git diff notebooks/01_collection.ipynb | grep -i "opendart.fss.or.kr/api" 결과 검토 — URL 파라미터에 key 없는지
  • ☐ response.url 직접 출력 셀 없는지
  • ☐ 에러 traceback에 key가 들어가지 않는지 (가능하면 try/except로 감싸기)
실수 발생 시 — 즉시 대응
  1. OpenDART에서 해당 key revoke + 새 key 발급
  2. 새 key로 .env 갱신
  3. git history rewrite (BFG 또는 git filter-repo) + force push (팀 합의 후)
  4. GitHub Secret Scanning 알림이 왔다면 "resolved" 처리

  • descriptor: 짧고 명확한 설명 (영문/숫자/-). 예: initial-pilot, rate-limit-v2, okt-only, tfidf-vocab10k

언제 archive 스냅샷을 만드나

✅ 스냅샷 해야 할 때

  • 전처리 파이프라인 큰 변경 (정규식 패턴 교체, 형태소 분석기 swap)
  • vocab 크기·임계값 등 파라미터 sweep 결과 보존
  • 회의에서 "이 결정 후 결과가 안 좋아지면 돌아가자" 합의된 시점
  • Phase 2 batch 시작 직전 (Phase 1 파일럿 보존)

❌ 스냅샷 안 해도 되는 때

  • 오타 수정, 변수명 변경, 주석 추가
  • 마크다운만 보강
  • 코드 한 줄 추가/제거 (이런 건 git history로 충분)
